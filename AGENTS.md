# Project Rules

## Workflow

- Always check the rendered page after HTML or CSS changes. Do not rely only on code review.
- For visual fixes, verify both desktop and mobile layouts in the browser before finishing.
- If something looks crooked, inspect the actual rendered block first and only then edit CSS.
- Do not leave local changes uncommitted at the end of the task.
- After meaningful changes, create a git commit and push to `origin/main` unless the user explicitly says not to.
- If a change was made but not pushed yet, state that clearly.

## Git

- Keep the repo clean.
- Commit every finished batch of changes.
- Push after commit so GitHub Pages can update.
- Do not claim a push happened unless `git push` actually succeeded.

## Project Context

- This site is for the user's mother and represents an atelier in Orsk.
- The site should clearly present both tailoring and clothing repair, not only repair.
- Write in plain Russian, without marketing fluff or template-like wording.
- Avoid visible TODO text, placeholder copy, or "later add this" phrasing in user-facing content.
- The site should help reduce repetitive questions from clients about services, pricing, and how to contact the atelier.

## Content Priorities

- Emphasize tailoring, repair, fitting, and alterations.
- Make phone, address, and contact flow obvious.
- Explain that pricing depends on complexity, fabric, fittings, and scope of work.
- Prefer concrete examples of services over vague general descriptions.
- If business details are missing, ask for the exact services, popular orders, hours, messengers, prices, and photos.
